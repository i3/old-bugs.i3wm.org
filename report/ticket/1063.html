Description

... but seriously, I think I found a color parsing bug. I have i3 installed on my x86_64 laptop and on a Rasberry Pi, both running Arch. (The pi runs Arch ARM) The following code:

set $blue  #00aaff
set $gray  #606263
set $white #ffffff
set $black #000000

# colorclass            border   backgr  text    indic
client.focused          $blue    $blue   $white  #2e9ef4
client.focused_inactive $white   $white  $black  $blue
client.unfocused        $white   $white  #a0a0a0 #292d2e
client.urgent           #2f343a  #900000 $black  #900000
client.background $white

makes blue active windows and white inactive ones as expected on the laptop, but on the Pi the active ones come out hot purple. i3status still handles them correctly. I'm not too familiar with hex color codes, so I can't tell right off what's going wrong, but I have noticed that commenting out some of those lines will restore the right colors to titlebars but leave window borders messed up and vice versa. So I think maybe two color values are being added together rather than one replacing the other? Or ARM has some different integer handling or something. Here's a table of what happens when I set $blue to random different colors:

# expected --> measured (by gcolor2)
#00aaff --> #ad5dff
#000000 --> #000000
#ffffff --> #ffffff
#e432a6 --> #315531
#669559 --> #9daace
#7a6116 --> #6320b5
#c53a57 --> #3949bd
