#1426 accepted defect

NET_ACTIVE_WINDOW messages steal focus

Reported by: i3-new-ticket Owned by: michael
Priority: major Component: general
Version: 4.8 Keywords:
Cc: Is it reproducible?: Always
git version: 4.8 (2014-06-15, branch "4.8") Operating system: Affects all
X11 program: Distribution / flavor:
Your IRC nick:

Description

Any client on the current workspace can steal focus with a NET_ACTIVE_WINDOW message. For example, if I have a terminal window and a chromium window open, and enter "chromium google.com" in the terminal, the terminal will lose focus to chromium. I would prefer that the terminal remains focused, but the chromium container could have its urgency flag set.

Change History (4)

comment:1 Changed 3 months ago by TonyC

  • Is it reproducible? set to Always
  • Operating system set to Affects all

I agree with this and would support this change.

I've never found it convenient that focus is set in this way. I think focus should be set only when a new window opens or the user takes some action that is intended to change focus.

I think that setting the window to urgent will accomplish everything that setting focus will. The application is usually just trying to get you to pay attention to the window in the message, and if you care, it's usually not any extra effort to set focus on the window yourself.

However, it's not really too inconvenient and it rarely causes problems for me. Also, this being a matter of style, there is room for disagreement.

comment:2 Changed 3 months ago by michael

  • Owner set to michael
  • Status changed from new to accepted

I think having these messages set focus is expected behavior, but I can see how opinions on that may be different.

Note that applications can also just set the focus, without sending the window manager a NET_ACTIVE_WINDOW message. Fixing this will hence not fix all instances of focus stealing. It may be reasonable to mitigate that by telling upstream authors to use NET_ACTIVE_WINDOW, though.

So, patches that introduce a configuration option for this behavior will be accepted.

comment:3 Changed 2 months ago by TonyC

I think I would rather see setting the window to urgent be the default. I think having a configuration option is exposing too much of ewmh internals in the config which runs the risk of bogging down the docs.

comment:4 Changed 2 months ago by michael

That’s a fair point. I don’t have a strong opinion on this one. I suggest that we introduce the urgent-instead-of-focus behavior to “next” after a new major release, then ask on the mailing list what people think about it.
