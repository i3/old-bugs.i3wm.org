# any combination of Num_lock, Caps_lock and Mode_switch

comment:19 Changed 5 years ago by michael

Just to be clear about why the current implementation is the way it is:

While Mode_switch is a modifier, the Mode_switch bit is never set when binding to it. This seems to be a bug in X, but using XKB, you can check the correct state (that is, if the Mode_switch bit is present). So i3 grabs the key itself and then checks for the Mode_switch bit itself.

As you noticed, this is far from ideal. If there is a better way to keep this functionality, I am happy to learn it.

comment:20 Changed 5 years ago by LemmingKing

I didn't have enough time to implement the patch with context-based bindings I described above this week.
But I've wrote a different patch (mode_switch_and_caps.patch):

  • caps-lock issue is removed (I suppose, Michael just forgot to write one more line in his grab_keycode_for_binding function).
  • now, Mode_switch-aware bindings are bound not with 0 modifiers mask, but rather with all their modifiers except for Mode_switch (which is a virtual modifier, created by Michael).

So now, "Mode_switch+Mod4+f" binding doesn't catch the bare 'f' key, so it doesn't have to forward it to clients then. So, while forwarding is still unfixed it produce less problems.

Please, give it a try.

NOTE: If you use permanent group switching (like in russian case) rather then modifier-based, you still have to write TWO bindings for each key:

# works in english layout
bindsym Mod4+f f
# works in alternative layout
bindsym Mode_switch+Mod4+f f

Also, in this case you can't use bindings like this:

bind Mode_switch+10 m1

because they will trigger switching workspace each time you type a number while alternative group is active.

BTW: I don't know if this is a case, but the whole this stuff seems to break down when you use more then 2 alternating groups.
