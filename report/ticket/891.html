Description

Thanks for your report. I copied your emacsclient file and tried it. As far as I can tell, i3-dmenu-desktop works fine:

  1. When completing emacsclient in i3-dmenu-desktop, it sends exec "emacsclient -c " to i3, which is the result of your Exec= line, without the %F. As per the specification, this is what should happen: “If the application should not open any file the %f, %u, %F and %U field codes must be removed from the command line and ignored.” (see ​http://standards.freedesktop.org/desktop-entry-spec/latest/ar01s06.html)
  1. When completing emacsclient, then adding a filename separated by a space (see i3-dmenu-desktop’s manpage), it works correctly: it sends exec "emacsclient -c \"/home/michael/i3/i3-dmenu-desktop\"" to i3 and a new Emacs window is opened with that file.

In fact, the emacsclient command, when run from a terminal, requires a filename. Therefore, I suggest using the standard emacs24.desktop for launching a new emacs instance and only using emacsclient when providing filenames.

comment:16 Changed 2 years ago by michael

Thanks for the strace output. I now know what was going on:

  • dmenu returns Emacsclient\n starting from version 4.4
  • i3-dmenu-desktop notices there is no entry for that and tries to interpret it as a command with a filename argument, therefore falls back to emacs.desktop with client\n as argument.

I fixed it by stripping the newline when interpreting choice. Thanks for your help on this.
