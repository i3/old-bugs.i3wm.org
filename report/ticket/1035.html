Description

Current state:

If there's multiple windows in scratchpad, scratchpad show cycles between them. If a specific window from scratchpad is shown, executing scratchpad show makes it disappear, same goes if you execute move scratchpad on that specific window, only after executing scratchpad show again, other window from scratchpad is shown. In short, no matter if you execute scratchpad show or move scratchpad on already shown scratchpad window, to make it disappear, executing scratchpad show again, will always yield next/other scratchpad window.

I'm addressing the problem of showing again with scratchpad show the same window, (a specific one,) previously shown.

Proposition:

If we would execute move scratchpad on a scratchpad window (which is already shown), instead of only make it disappear, we would make it again the first in scratchpad show cycle. We use already existing command (so no new key-binding) and at the same time we solve the-same-action-performing of scratchpad show and move scratchpad on already shown scratchpad window.

Being not familiar with i3's internals, I'm not sure if it's achievable, though.

Your work is appreciated, thank you.

Philippe took a stab at this in ​http://cr.i3wm.org/patch/248

The patch made it clear to me that far too much needs to be changed to accomodate this (somewhat trivial, in the sense that the implications are trivial) feature request.

Thinking about it again, I take the position that if you really benefit from the cycling order in the scratchpad, you are abusing it. It is supposed to either:

  • be a place for a single useful window that you need on multiple workspaces, e.g. a Python REPL
  • same usecase, but with a set of fixed windows which you can identify using criteria (even this second usecase is only present because it is a result of our code structure, it was not intended)

Anything more than that is not supported — after all, a tiling window manager such as i3 gives you plenty of other ways to organize your windows.
