Layout modes do not persist in a workspace

Description

I have two windows, say xterms, side-by-side in a workspace, say 9. I then switch to tabbed layout mode. So far, so good. Then I close both windows, but I don't leave workspace 9, so its little tab in the status bar is still present (if I did go to another workspace, the little tab for workspace 9 would disappear in the status bar). If I then open two new xterms in workspace 9, they will be arranged in the default layout, not the tabbed layout which was in force with the prior set of xterms.

I think from a UI standpoint, this behavior is unfortunate. When I set the layout of workspace 9 to 'tabbed', I expect that to persist, even if workspace 9 becomes temporarily childless. The current behavior surprised me, so invoking the Principle of Least Surprise, I would advocate fixing this. You already allow assignment of window classes to workspaces that don't yet exist, so talking about non-existent workspaces is already in the i3 vocabulary -- there must be a table built internally from the 'assign' statements in the config file that maps classes to workspaces. So why not have another such table that maps workspace to current layout. In effect, I'm asking that workspace_layout become per-workspace, a vector rather than the current scalar, and that it be managed dynamically, recording whatever the current layout is for each workspace. The workspace_layout configuration command could be extended to take an additional argument, the workspace number, e.g.,

workspace_layout tabbed 1
workspace_layout default 2

The internal table should be initialized to what appears in the config file, with the default being 'default' for unspecified entries.
