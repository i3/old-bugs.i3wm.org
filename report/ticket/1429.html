Description

Description

​This commit allows actions to be bound to the mouse keys; however, it is implemented in such a way that it only works if the mouse pointer is currently on container (unless a modifier key is used).

Although I understand the reason behind that for the primary mouse buttons, it is inconvenient for some of the additional mouse buttons. In particular, more and more laptops these days have multi-touch functionality which register as buttonN.

I my case, three- and four-finger swipes register as button8 through to button15 (depending on the direction). I wanted to bind the three-finger swipes left and right to moving to the prev/next workspace:

bindsym button10 workspace prev_on_output
bindsym button11 workspace next_on_output

but these bindings are essentially useless since I can only use them if the cursor is over a window title.

Desired Behaviour

It would be good if the non-primary buttons did not require for the cursor to be on a container, though the inconsistent behaviour between primary and non-primary button could be annoying (and may not fix this for everyone).

Alternatively, having the possibility to toggle on/off that behaviour when declaring the binding would be very helpful. Something like

bindsym [Modifiers+]button[n] [everywhere|containers] command

could be good.

Temporary Solutions

In case someone finds have the same issue and finds this ticket, it is possible to use xbindkeys in order to circumvent the issue in i3. In ~/.xbindkeysrc , add

"i3 <command>
    b:N

So in my particular case, I added

"i3 workspace prev_on_output"
    b:10

"i3 workspace next_on_output"
    b:11

This ticket was fixed in commit ​http://c.i3wm.org/74b69d6d:

Add mouse binding pointer position configuration

Add the `--whole-window` switch for mouse bindings. This switch controls
what part of the container the pointer must be over to trigger a mouse
binding. The default is to only trigger mouse bindings over the
titlebars. With this switch, a mouse binding will be triggered over the
main part of the window as well.

This is a breaking change to the previous behavior, which would trigger
a mouse binding with a modifier over any part of the window.
