Description

Moving the mouse between two workspaces on separate screens can cause the cursor to be placed on the center of the first screen instead of moving to the second one. Happens regularly while moving from firefox container to any other window. Disabling focus following mouse is a workaround.

comment:9 follow-up: ↓ 10 Changed 2 years ago by gotterdammerung@…

I tried debugging this last night.

The erroneous cursor warping is triggered if you move the focus quickly back
and forth between workspaces. Shellscript to reproduce follows.

I think it is a race condition between x_set_warp_to() (x.c:1070) and the call
to xcb_query_pointer() in x_push_changes() (x.c:846).

x_set_warp_to() stores the rectangle to warp the pointer to if needed.
xcb_query_pointer() gets the current mouse coordinates.
Warping to the center of the rectangle happens if it lies on a different Output than
the mouse cursor. (x.c:921)

So the pointer is on screen 1 and x_set_warp_to() is called.
Before x_push_changes() runs, I move the cursor to screen 2.
x_push_changes() runs. The Outputs are different now so the mouse gets
warped to screen 1, or more exactly to the center of the container it came from.

/* If we need to warp later, we request the pointer position as soon as possible */
This comment (x.c:850) kinda gave it away. It has 'race condition' written all
over it.

I think the problem here could be that the x events (motion, enter, etc.) are handled
serially and contain pointer coordinates which were valid for the time of the
event, whereas xcb_query_pointer asynchonously fetches the current pointer
coordinates.

Line numbers are from current master (29c027db3be1bea34bcc2a23bf59ca7ddf986873).

comment:14 Changed 2 years ago by gotterdammerung@…

My answer is: "No."

But...
If you enable 'focus follows mouse' and move the focused window via Meta+Shift+<n> to a workspace (or just focus another workspace via Meta+<n>) which is on another screen and do _not_ warp the mouse onto the other screen, you get a state which is a special case: the cursor is on your first screen, but the focus is on another screen.
The user has to click once, move the mouse to the title bar or move the mouse to another window to restore the 'my focus is where my mouse is'-consistency.

i3 could automatically focus the window on which the pointer remains: that would make you lose focus through moving the first window, which is bad. Ion3, openbox and icewm reportedly do this.

Another option is to warp the mouse to the window's new position: If you see the mouse cursor as an extension of the users hand, this is like slapping the user on the hand, which is worse. This is what i3 tried to do.
That's the only reason i see for warping the pointer.

My fix (doing nothing) totally works for me, even in the case described above. I just moved a window to the right screen, it's logical i have to move the mouse too, if i want to click on it.

The i3 developers should select an option and implement it cleanly.
Nice but not neccessary: let the user decide: 'mouse_follows_focus' and 'strict_mouse_focus' comes to mind.

comment:22 Changed 10 months ago by TonyC

Ok good. This bug was fixed by these two commits:

​http://code.stapelberg.de/git/i3/commit/?h=next&id=8ece9950012e0a2d20db9d4d15bc3a3e30b96121

Bugfix: prevent unneeded render on pointer enter

When `focus_follows_mouse` option is on, prevent an uneeded render on
pointer enter when the con is already focused.

This pointer enter might be caused by entering a window decoration of an
already-focused container.

This extra render can cause concurrency issues when focus is set
asynchronously with WM_TAKE_FOCUS.

​http://code.stapelberg.de/git/i3/commit/?h=next&id=9cee8dac5e9057bf2cb72edc3cf398fdc08b8d73

Bugfix: Set input focus with last timestamp

Setting input focus with the last event timestamp prevents race
conditions when setting focus.
