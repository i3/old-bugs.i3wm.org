It's possible to bring up i3bar while i3lock is active

Reported by: mail+i3@… Owned by: michael
Priority: major Component: general
Version: 4.4 Keywords:
Cc: Is it reproducible?: Always
git version: 4.4-15-g2a18e9d (2012-12-17, branch "next") Operating system: Affects all
X11 program: Distribution / flavor:
Your IRC nick:

Description

With i3lock active and i3bar set to hide, the bar can be shown by entering a wrong password and pressing $mod. This should obviously not happen.

It's not possible - fortunately - to do anything beyond this, eg start dmenu or interact with tray icons. I don't think it's very relevant security-wise, though it does leak a bit of system information.

Change History (6)

comment:1 Changed 2 years ago by michael

  • Is it reproducible? set to Always
  • Operating system set to Affects all
  • Status changed from new to accepted

This is true.

The problem is that the libpam library call blocks for a long time to prevent brute-forcing too many passwords. Since i3lock is single-threaded, the entire program will block and thus it will not try to gain the upper hand when another window is mapped on top of it.

I wanted to check whether one could “fix” this in i3 by lowering windows below i3lock as long as it runs, but I think that requires a non-standard X11 hint.

comment:2 Changed 2 years ago by michael

Given that programs such as dunst can just call ConfigureWindow to raise their X11 windows without i3 noticing any of that (dunst is override_redirect=1), there doesn’t seem to be a clean solution to this.

The only thing I can think of is moving the password verification into a separate process/thread.

comment:3 Changed 19 months ago by anonymous
